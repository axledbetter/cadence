# v7.12.0 — Stack template expansion + user-type profiles

## Goal

Two related additions that broaden claude-autopilot's reach:

**Bucket A — Stack template expansion.** Add five new example specs under
`examples/specs/` so that `claude-autopilot examples <stack>` can scaffold
Rails, Laravel, Spring Boot, .NET Web API, and Swift CLI projects. The
existing v7.10.1 examples verb (Node, Python, FastAPI, Go, Rust) ships
five — this doubles the catalog to ten and unblocks users on the major
back-end ecosystems we currently turn away.

**Bucket B — User-type profiles.** Introduce a profile mechanism that
adapts autopilot behavior to the team shape: solo, small-team,
oss-maintainer, enterprise, learning. A profile is a thin YAML overlay
that adjusts codex-pass counts, auto-merge default, mandatory frontmatter,
pause points, audit logging, and the PR template. The active profile is
resolved from (in precedence order, lowest to highest) the shipped
default → `.autopilot/profile` repo file → `CLAUDE_AUTOPILOT_PROFILE`
env var → `--profile` CLI flag. The skill reads the resolved profile in
Step 0 and adjusts the pipeline accordingly.

The two buckets share a release because both are content-heavy + zero
runtime risk to existing flows when defaults stay solo / no profile file
exists. The version is **v7.12.0** to avoid collision with the in-flight
v7.11.0 concurrent-dispatch spec (PR #187, issues #188–193).

## Non-goals

- **Hosted profile sharing** (team-wide profile registry) — separate spec.
- **Profile inheritance** (e.g. `small-team` extends `solo`) — separate spec;
  v7.12.0 ships flat profiles only.
- **Per-environment profiles** (dev vs prod profile selection) — separate spec.
- **AI-generated custom profiles** from `claude-autopilot init --interview` — separate spec.
- **iOS app scaffolding** — Swift in this release is CLI-only (SwiftPM +
  XCTest). iOS apps require Xcode-generated `.xcodeproj` bundles that don't
  shell-script reliably; deferred.
- **Gradle-based Spring Boot** — the example uses Maven for reproducibility
  with `mvn -B`; a Gradle variant can land separately if asked.
- **Replacing the existing risk-tier policy** for solo. The default profile
  preserves v7.10.1 behavior exactly; existing users see no change unless
  they opt in.
- **Live profile editing UI / interactive switcher** beyond `init`'s prompt.
  Power users edit `.autopilot/profile` directly.

## Bucket A — Stack templates

### Shape of each spec

All five new specs follow the existing template shape (cross-reference
`examples/specs/node-cli.md` for the canonical example):

- `# <project-name> — <stack one-liner>` header.
- `## Goal` — 1–2 paragraphs describing what the user is building and
  what scaffold pattern it demonstrates. Mention the build tool +
  test framework explicitly.
- `## Files` — bulleted list. Each bullet is `` `path` — short role
  description ``. Scaffolder-readable: it parses bullets, treats
  backticked tokens as paths, and uses the prose to decide template
  contents.
- `## How to use` — pipe-friendly command block showing
  `claude-autopilot examples <stack> > spec.md` followed by
  `claude-autopilot scaffold --from-spec spec.md` and the stack's
  native test/run command.

Target length: ~100–150 lines each, matching the existing five.

### Scaffolder compatibility

The v7.10.1 scaffolder is intentionally template-agnostic: it reads
`## Files` bullets, parses backticked paths, and dispatches to the
generic file-writer with stub content. For these five new stacks the
scaffolder produces an empty file at each declared path with a
`// TODO: implement` (or language-appropriate) header comment.

**No new language-specific codegen ships in v7.12.0.** Filling the
files with working content is the impl-agent's job, as it is for the
existing five. This keeps the spec PR small and the diff predictable.

**Scaffold-vs-working-code clarification (revised pass 1, finding #9):**
each new example's `How to use` section opens with a one-line note that
`claude-autopilot scaffold --from-spec` writes the file layout only;
native commands like `bundle exec rspec`, `dotnet test`, `swift test`,
etc., are expected to fail until the implementation agent fills the
bodies. The existing five example specs get the same one-line note in
the same v7.12.0 PR for consistency.

## Bucket B — User-type profiles

### Profile mechanism overview

A profile is a small YAML file at `presets/profiles/<name>.yaml` that
overlays autopilot defaults. The five shipped profiles are
`solo`, `small-team`, `oss-maintainer`, `enterprise`, `learning`.

**Directory layout (revised pass 1, finding #3):** PR templates live
under `presets/profiles/templates/` rather than at the same depth as
profile YAMLs, so that "available profiles" is unambiguously the set
of `*.yaml` stems directly under `presets/profiles/`. The
`pr_template_path` key in each profile resolves **relative to the
package root** — i.e. the canonical value is
`presets/profiles/templates/small-team-pr-template.md`, NOT
`presets/profiles/templates/small-team-pr-template.md` (revised pass 2, WARN #2).
The profile-table summary below reflects this canonical form. A
resolver unit test asserts each shipped profile's `pr_template_path`
resolves to a file that exists in the package root.

### File format

```yaml
profile: small-team
description: 2–10 dev team with human PR review and branch protection.
codex_passes:
  low: 0
  medium: 1
  high: 2
auto_merge: false
require_risk_frontmatter: false
pause_at_steps: []           # autopilot step numbers, e.g. [4, 6, 9]
audit_log_path: null         # null = disabled; '.claude/audit/' = enabled
codex_explanations: false    # false = raw codex output; true = LLM-rewrap with "why this matters" (additive — preserves raw severity/title/suggestion)
pr_template_path: presets/profiles/templates/small-team-pr-template.md
contributor_policy:          # null when not applicable; populated for oss-maintainer (pass 1 finding #2)
  external_high_codex_passes: 3
  membership_provider: github-org
```

All five profile files plus their PR templates (where applicable) ship
in the npm package. The keys are an **exhaustive overlay surface**:
adding new keys later is a minor-version bump and requires a schema
update.

### Resolution and precedence

Active profile resolves in this order (lowest to highest precedence):

1. **Shipped default**: `solo` (hard-coded constant).
2. **Repo file**: `.autopilot/profile` — a strict single-line file
   containing the profile name (e.g. `enterprise\n`). **Parsing rules
   (revised pass 1, finding #7):** trailing whitespace and a single
   trailing newline are tolerated; any other content (multiple
   non-empty lines, embedded whitespace, comments) raises a typed
   `ProfileResolutionError` instead of being silently first-line-parsed.
3. **Environment variable**: `CLAUDE_AUTOPILOT_PROFILE=<name>`.
4. **CLI flag**: `--profile <name>` (global flag, applies to any verb).

Each layer overrides the previous. If `.autopilot/profile` names a
profile that doesn't exist in `presets/profiles/`, fail at resolution
time with a clear error listing the available profile names — do NOT
silently fall back to `solo` (would mask typos in production
configurations). The same rule applies to the env var and flag.

### Schema validation

`presets/schemas/profile.schema.json` enforces the YAML structure
(JSON Schema draft-07). Required keys: `profile`, `description`,
`codex_passes` (with all three sub-keys: `low`, `medium`, `high`).
Optional keys default to safe values:

| Key | Default | Type |
|---|---|---|
| `auto_merge` | `true` | boolean |
| `require_risk_frontmatter` | `false` | boolean |
| `pause_at_steps` | `[]` | integer[] (each in `[0, 10]`) |
| `audit_log_path` | `null` | string \| null |
| `codex_explanations` | `false` | boolean |
| `pr_template_path` | `null` | string \| null |
| `contributor_policy` | `null` | object \| null (see below) |

Schema enforces (JSON Schema draft-07 — document-level constraints
only; filename ↔ `profile` matching is a separate resolver-level check,
see below):

- **`additionalProperties: false` at the top level AND on every
  nested object** (`codex_passes`, `contributor_policy`) — revised
  pass 2, NOTE #1. Adding a new key requires a schema update; no
  silent acceptance of unknown fields.
- `codex_passes.{low,medium,high}` are integers in `[0, 5]`.
- `pause_at_steps` integers in `[0, 10]`.
- `audit_log_path`, when non-null, must be a relative path inside the
  repo (rejects absolute paths and `..` escapes to avoid writing
  outside the repo).
- `contributor_policy`, when non-null, must contain
  `external_high_codex_passes` (integer in `[0, 5]`) and
  `membership_provider` (currently only `"github-org"` is accepted;
  reserved for future providers).

**Filename ↔ `profile` field match (revised pass 1, finding #4):**
because JSON Schema does not have access to external filename
context, this constraint is enforced by `resolveProfile` itself.
After loading and JSON-Schema-validating the YAML, the resolver
asserts `config.profile === <filename-stem>` and raises a typed
`ProfileFilenameMismatchError` if they disagree. This is covered by
its own test case, independent of schema-validation tests.

**Validation-failure policy (revised pass 1, finding #6):** resolver
errors (unknown profile name, filename mismatch, YAML parse error,
schema violation) ALWAYS fail commands that need a profile —
including `autopilot`, `init`, `run`, `validate`, `examples`, etc.
`claude-autopilot doctor` runs the same validation and exits non-zero
by default on any failure. `--strict-profile` is reserved for
upgrading future deprecation warnings (e.g. obsolete keys, removed
profile names) to errors; it has no effect on the always-failing
hard checks above. This single policy replaces the earlier ambiguous
"aborts subsequent commands until corrected" wording.

### The five shipped profiles

| Profile | codex_passes (L/M/H) | auto_merge | require_risk | pause_at_steps | audit_log | explanations | contributor_policy | pr_template |
|---|---|---|---|---|---|---|---|---|
| solo | 1 / 2 / 3 | true | false | [] | null | false | null | null |
| small-team | 0 / 1 / 2 | false | false | [] | null | false | null | `presets/profiles/templates/small-team-pr-template.md` |
| oss-maintainer | 1 / 2 / 2 | false | false | [] | null | false | `{ external_high_codex_passes: 3, membership_provider: "github-org" }` | `presets/profiles/templates/oss-maintainer-pr-template.md` |
| enterprise | 3 / 3 / 3 | false | true | [] | `.claude/audit/` | false | null | `presets/profiles/templates/enterprise-pr-template.md` |
| learning | 1 / 2 / 3 | false | false | [4, 6, 9] | null | true | null | null |

**Specifics:**

- **solo**: default. Behaviorally identical to v7.10.1; existing users
  not opting into a profile see no change.
- **small-team**: humans review PRs, so codex passes are lighter. Auto-merge
  off — PRs await human approval. PR template prompts for reviewer
  assignment and links to the team's review checklist.
- **oss-maintainer**: external contributor PRs get an extra pass per the
  `contributor_policy` overlay. PR template includes a
  "First-time contributor?" section.

  **Membership check semantics (revised pass 1, finding #8):**
  - **Target org** is read from the GitHub repository's `owner.login`
    (resolved via `gh api repos/{owner}/{repo}` once per run and cached).
    If `owner.type === "User"` (user-owned, not org-owned), the
    membership check is skipped and `external_high_codex_passes` does
    NOT apply — there is no org to belong to. Emit a one-line
    diagnostic.
  - **Provider**: `membership_provider: github-org` calls
    `gh api orgs/<org>/members/<user>`. Result classification:
    - HTTP 204 → member; use the base `codex_passes.high`.
    - HTTP 404 → confirmed external contributor; bump high-tier to
      `external_high_codex_passes`.
    - HTTP 401 / 403 → auth or scope insufficient; do NOT bump. Emit
      a warning to the run log and proceed with the base `codex_passes`.
    - HTTP 429 / 5xx / network error → transient; retry once with
      exponential backoff (1s, then 4s), then do NOT bump and emit
      a warning. Never bump on uncertainty.
    - `gh` binary missing → do NOT bump; emit a warning.
  - **GitHub Enterprise**: respected via `gh`'s standard
    `GH_HOST` / `GH_ENTERPRISE_TOKEN` environment, since the call is
    routed through `gh api`. No special handling needed.
  - **Forks (revised pass 2, WARN #7).** The BASE repository's owner
    determines the org to check (`<org>` = base repo `owner.login`).
    The PR author's login is the subject being checked
    (`<user>` = PR author). The head repository's owner is irrelevant
    except as source metadata. An org member opening a PR from their
    personal fork still passes the membership check, because the
    `gh api orgs/<base-org>/members/<pr-author>` call only cares
    about the membership relationship, not which fork the PR came
    from. Tests cover: (a) org-owned base + personal fork by
    confirmed org member → no bump, (b) org-owned base + personal
    fork by non-member → bump, (c) personal-owned base repo
    (contributor_policy.membership_provider not applicable) → no
    bump regardless of author.
- **enterprise**: every tier gets 3 codex passes. `risk:` frontmatter
  is mandatory — autopilot refuses to start a spec without it. All
  codex findings + bugbot triage decisions + merge events append to
  `.claude/audit/<run-ulid>.jsonl` as immutable JSONL (append-only,
  one event per line; never rewritten or deleted by autopilot — log
  rotation/archival is the user's responsibility via standard ops
  tooling).
- **learning**: pauses for explicit user confirmation after Step 4
  (validate), Step 6 (push + PR), and Step 9 (merge). Codex findings
  are followed by a second LLM pass that adds "why this matters"
  prose. **Note**: this is the ONE profile that violates the
  "no mid-pipeline pause" CRITICAL invariant in `skills/autopilot/SKILL.md`.
  The skill text gets a carve-out: "If the active profile sets
  `pause_at_steps`, those pauses are an intrinsic part of the
  profile's contract and do NOT count as mid-pipeline check-ins."

### Profile resolution module

New file: `src/core/profile/resolver.ts`. Exports:

```ts
export interface ProfileConfig { /* schema mirror */ }

export function resolveProfile(opts: {
  cwd: string;
  envProfile?: string;
  flagProfile?: string;
}): { name: string; config: ProfileConfig; source: 'default' | 'file' | 'env' | 'flag' };

export function loadProfileByName(name: string): ProfileConfig;
```

The resolver:

1. Reads `<cwd>/.autopilot/profile` if it exists; trims; uses as
   candidate profile name. Empty/whitespace-only file is treated as
   absent (falls through to env/flag/default). Multiple non-empty
   lines, embedded whitespace, or comments raise
   `ProfileResolutionError` (revised pass 1, finding #5).
2. Overlays env var (`CLAUDE_AUTOPILOT_PROFILE`) if set. Empty or
   whitespace-only env values are treated as unset (revised pass 2,
   note #2).
3. Overlays flag (`--profile`) if set. Empty flag value
   (`--profile ""`) is rejected with a typed error (revised pass 2,
   note #2).
4. **Path-safety validation (revised pass 2, CRITICAL).** Before any
   file read: enumerate available profile stems by `readdir`-ing
   direct `*.yaml` entries under `presets/profiles/` (no recursion;
   skip the `templates/` subdirectory). The requested name MUST
   exactly match one of those enumerated stems. Independently, the
   name MUST also match the regex `^[a-z0-9][a-z0-9-]*$`. Names
   containing `/`, `\`, `..`, leading dots, or path separators are
   rejected with `ProfileResolutionError` even if the regex passes.
   This blocks path-traversal attacks via malicious
   `.autopilot/profile` content or env var injection.
5. Loads the validated name's YAML from `presets/profiles/<name>.yaml`.
6. Validates against `presets/schemas/profile.schema.json`. Filename ↔
   `profile` field match is enforced HERE by the resolver, not by the
   schema (schema cannot access external filename context — revised
   pass 1, finding #3 + pass 2, WARN #5).
7. Returns config + source for diagnostics.

Failures at any step throw a typed `ProfileResolutionError` with a
single-line user message and a remediation hint. **Hard failures
(unknown name, path-traversal attempt, schema violation, filename
mismatch, YAML parse error) fail commands unconditionally** — the
`--strict-profile` flag is reserved exclusively for future deprecation
warnings, NOT for gating hard failures (revised pass 2, WARN #4).

**Required tests for path safety (revised pass 2, CRITICAL):**
- `.autopilot/profile` containing `../solo` → rejected
- `.autopilot/profile` containing `/tmp/x` → rejected
- `.autopilot/profile` containing `templates/foo` → rejected
- `.autopilot/profile` containing `solo.yaml` (with extension) → rejected
- `CLAUDE_AUTOPILOT_PROFILE=../../etc/passwd` → rejected
- `--profile ..` → rejected
- `--profile ""` → rejected
- `CLAUDE_AUTOPILOT_PROFILE=""` → treated as unset

**Profile-resolution-required vs profile-resolution-optional
commands (revised pass 2, WARN #6).** Resolver only runs for commands
whose behavior depends on the active profile: `autopilot`, `run`,
`validate`, `pr`, `merge-or-queue`. Commands that do NOT depend on
profile behavior skip resolution entirely so a malformed
`.autopilot/profile` doesn't block remediation:
- `examples` (pure scaffold output)
- `profile list` (enumerates available, not active)
- `init` detects an invalid existing `.autopilot/profile` and offers
  to overwrite rather than failing before the interactive prompt
- `doctor` and `profile show` ARE strict — they exist to surface
  exactly these problems and must fail loudly on them

### CLI changes

- **`claude-autopilot init`** — adds one new interactive prompt after
  the existing setup steps:

  > Which user profile fits your team?
  > [1] solo          — single dev, fast iteration (default)
  > [2] small-team    — 2–10 devs, human PR review expected
  > [3] oss-maintainer — community contributions, contributor-aware policy
  > [4] enterprise    — strict review, mandatory risk frontmatter, audit log
  > [5] learning      — slower pace, verbose explanations, pauses between steps
  > Choice [1]:

  Writes the selected name to `.autopilot/profile`. Skippable with
  `--non-interactive` (defaults to solo) and `--profile <name>` (pre-fills).
- **Global `--profile <name>` flag** — accepted by every verb. When
  present, overrides file + env. Reported in `claude-autopilot doctor`.
- **`claude-autopilot profile show`** — prints active profile name,
  source (`default` / `file` / `env` / `flag`), and full resolved
  overlay as YAML to stdout. Exit 0 on success.
- **`claude-autopilot profile list`** — prints available profile names
  one per line, derived from `*.yaml` stems directly under
  `presets/profiles/` (subdirectories such as `templates/` are NOT
  scanned — revised pass 1, finding #3). Exit 0. A unit test asserts
  the output is exactly the five shipped profile names.
- **`claude-autopilot doctor`** — extended to validate the resolved
  profile against the schema, check that `.autopilot/profile`'s
  content names a known profile, and report the active profile in its
  summary block. Hard profile errors (unknown name, path-traversal
  attempt, schema violation, filename mismatch, YAML parse error)
  fail `doctor` unconditionally with exit code non-zero — there is
  no flag that gates these failures. The `--strict-profile` flag is
  reserved for FUTURE warnings that should escalate to failures in
  CI (no warnings exist yet; the flag is a no-op in v7.12.0 and
  surfaces a "no warnings in this version" message). Revised pass 2,
  WARN #4.

### Skill content changes

`skills/autopilot/SKILL.md` learns to read the resolved profile in
Step 0 (before any LLM call) and adjust the pipeline per the overlay:

- **`codex_passes`**: overrides the risk-tier defaults baked into the
  existing "Codex pass policy" table. The risk tier still determines
  which bucket (`low`/`medium`/`high`) is read; the profile sets the
  count within that bucket. If `codex_passes.<tier>` is `0`, codex is
  skipped entirely for that tier and the skill emits
  `[autopilot] Codex skipped (profile=<name>, tier=<tier>, count=0)`
  to the run log.
- **`auto_merge: false`**: Step 9's admin-squash-merge is replaced
  with "PR is ready for human review at <URL>. Autopilot exits;
  re-invoke `claude-autopilot run resume <ulid>` after merge to run
  any post-merge hooks." The run-state engine records the run as
  `paused_at_review` (terminal-from-autopilot's-perspective but
  resumable).
- **`require_risk_frontmatter: true`**: preflight (between Step 0
  approval and Step 1 dispatch) checks the spec's YAML frontmatter
  for a `risk:` key. Missing or invalid value aborts with
  remediation hint. The keyword auto-escalation fallback (used by
  solo) is **disabled** under this profile.
- **`pause_at_steps`**: after each listed step completes, the skill
  emits a structured pause: prints the step's outcome summary, prints
  "Press [c] to continue, [a] to abort, [d] to dump state and exit
  for inspection.", and waits on stdin. Re-runnable via
  `claude-autopilot run resume <ulid>` if the user exits.
- **`audit_log_path`**: when set, every codex finding (raw JSON),
  bugbot triage decision, validate outcome, merge event, and pause
  resolution is appended as one JSONL line to
  `<audit_log_path>/<run-ulid>.jsonl`. Schema for each line:
  `{ "ts": <iso>, "event": <type>, "run_id": <ulid>, "step": <n>,
  "data": { ... } }`.

  **Filesystem safety (revised pass 1, finding #5):**
  - The audit-log writer resolves the real path of the target
    directory (via `fs.realpathSync`) AFTER ensuring it exists, then
    asserts the realpath is under the repo root. Symlink-escape
    (e.g. `.claude/audit -> /tmp/outside`) is rejected with a typed
    `AuditPathEscapeError`; autopilot aborts the run rather than
    writing outside the repo.
  - Each per-run log file is opened with `O_CREAT | O_APPEND |
    O_WRONLY` and `O_NOFOLLOW` on the file itself (so a pre-existing
    symlinked log file at the target path also fails). If the file
    already exists as a regular file (e.g. resume of a prior run),
    append mode preserves prior content — autopilot never seeks,
    rewrites, or truncates the audit log.
  - **Write atomicity (revised pass 2, WARN #8).** `PIPE_BUF` only
    applies to pipes and FIFOs, not regular files, so it cannot be
    cited as a regular-file guarantee. The actual contract is: each
    JSONL event is encoded into a single `Buffer` and emitted via a
    single `write()` call against an `O_APPEND`-opened file
    descriptor. `O_APPEND` makes the kernel set the file position
    atomically before each write, so concurrent appenders within a
    single process (or across processes) never overlap at the byte
    level for that single write. The implementation MUST NOT split a
    single JSONL line into multiple `write()` calls (no streaming
    JSON.stringify into a write stream; buffer fully first).
  - **Cross-process strictness.** For best-effort multi-process
    non-interleaving (e.g. multiple autopilot CLI invocations sharing
    a repo), the above is sufficient on POSIX. Strict guarantees
    (Windows, network filesystems where `O_APPEND` semantics are
    unreliable) would require an additional `flock`-based advisory
    lock; this is OUT OF SCOPE for v7.12.0 and documented as a
    known limitation.
  - **Required test:** a multi-process test (NOT just async-in-one-
    process) spawns two `node` workers that each append 100 JSONL
    events to the same audit log path concurrently. The test asserts
    every line in the resulting file parses as valid JSON (no
    interleaved partial writes).
- **`codex_explanations: true`**: after each codex pass, the captured
  findings (CRITICAL/WARN/NOTE) get a single follow-up LLM call that
  ADDS a `why_this_matters` field to each finding, using the same
  provider/model that the host project's codex-review script uses.
  **Additive-only (revised pass 1, finding #10):** the original
  `severity`, `title`, `description`, and `suggestion` fields are
  preserved verbatim — the explanation pass MUST NOT modify them.
  Both the raw findings and the expanded findings are written to
  the run log (`.claude/run-state/<run-ulid>/codex-pass-N.json` —
  this is the existing per-run telemetry path, written regardless of
  `audit_log_path`) so the user can always diff the explanation
  against the raw output even when full enterprise audit logging is
  disabled.

The skill's preamble gets a one-paragraph "Active profile" block at
the top, printed once per run:

```
[autopilot] Active profile: small-team (source=file)
[autopilot] codex_passes: low=0 medium=1 high=2 | auto_merge=off | pauses: []
```

## Testing strategy

### Unit tests (fast, no shell)

- **`profile/resolver.test.ts`**:
  - Default fallback (no file, no env, no flag → `solo`).
  - File-only resolution.
  - Env overrides file.
  - Flag overrides env + file.
  - Unknown profile name → typed error, lists available names.
  - **Parse-rule cases (revised pass 2, WARN #3) — ALLOW:**
    - `enterprise\n` (canonical)
    - `enterprise   \n` (trailing whitespace tolerated)
    - `enterprise\n\n` (single trailing blank line tolerated)
  - **Parse-rule cases (revised pass 2, WARN #3) — REJECT:**
    - `enterprise\nsolo\n` (multiple non-empty lines)
    - `enterprise # comment` (no inline comments)
    - `enter prise` (embedded whitespace)
    - empty file or whitespace-only file (treat as unset, fall through to env/flag/default — distinct from "reject")
  - **Path-safety cases (revised pass 2, CRITICAL):**
    - `../solo` → rejected with `ProfileResolutionError`
    - `/tmp/x` → rejected
    - `templates/foo` → rejected
    - `solo.yaml` (with extension) → rejected
    - `CLAUDE_AUTOPILOT_PROFILE=../../etc/passwd` → rejected
    - `--profile ..` → rejected
    - `--profile ""` → rejected
    - `CLAUDE_AUTOPILOT_PROFILE=""` → treated as unset (NOT rejected)
  - **Resolver-level checks moved out of schema (revised pass 2, WARN #5):**
    - Mismatched `profile` key vs. filename → `ProfileFilenameMismatchError`
    - PR template path resolves to a file that does not exist → typed error
    - All shipped profiles' `pr_template_path` values resolve to real files (one assertion per profile)
  - Source tag (`default` / `file` / `env` / `flag`) is accurate in each case.
- **`profile/schema.test.ts`** (schema-only constraints; filename
  match lives in resolver.test.ts per revised pass 2, WARN #5):
  - All five shipped profiles validate against the schema.
  - Synthetic invalid profiles fail validation with the offending
    field surfaced:
    - wrong types (e.g. `codex_passes.low: "two"`)
    - out-of-range integers (e.g. `codex_passes.high: 5`)
    - absolute `audit_log_path` (e.g. `/etc/audit`)
    - `..` escape in `audit_log_path` (e.g. `../etc`)
    - **unknown keys at the top level (revised pass 2, NOTE #1)**:
      `foo: bar` fails validation (`additionalProperties: false`)
    - **unknown keys in nested objects (revised pass 2, NOTE #1)**:
      `codex_passes.experimental: 1` fails validation
      (`additionalProperties: false` on every nested object)
    - **NOT IN SCHEMA TESTS:** mismatched `profile` key vs filename
      lives in resolver.test.ts, NOT here.
- **`profile/audit-log.test.ts`**:
  - Writer appends one JSONL line per event.
  - Concurrent appenders interleave but every line is a complete JSON object.
  - Never seeks/truncates the file (verified by stat'ing the file after a
    sequence of appends and confirming size monotonically increases).

## Risk classification

**Medium.** Two codex passes mandated.

- Bucket A is pure content + a lookup-table extension. Zero runtime
  risk to existing flows.
- Bucket B introduces a new resolution path and gates the skill's
  behavior. The risky bits are the unfamiliar pause-at-steps carve-out
  (violating an existing CRITICAL invariant in the skill), the
  oss-maintainer contributor-membership check (a new external API
  dependency: `gh api orgs/<org>/members/<user>` — must degrade
  gracefully if rate-limited or unauthorized), and the audit-log
  open-append semantics (must NOT lose events on crash).
- Not high-tier: no new secrets, no multi-tenancy, no auto-merge
  expansion, no repo mutation patterns. The default profile is a
  no-op for existing users.
