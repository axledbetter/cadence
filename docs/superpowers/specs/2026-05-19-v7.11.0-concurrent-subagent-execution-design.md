# v7.11.0 — Concurrent subagent task execution

## Goal

Replace the sequential one-task-at-a-time dispatch in autopilot Step 3 with a parallel scheduler that runs independent plan tasks concurrently in isolated worktrees, then merges results back into the feature branch in dependency order.

For a typical 5-task plan where 3 tasks are independent, this should cut Step 3 wall-clock by 1.5-2x. The ceiling is bounded by the longest dependency chain plus the slowest task on that chain.

## Non-goals

- Distributed execution across machines. All subagents run on the same host.
- Parallelizing within a single task. A task is the atomic unit; if it's too big, split it in the plan.
- Auto-detection of dependencies from code analysis. Dependencies come from explicit `depends_on:` annotations in the plan, with file-touch overlap as a fallback heuristic.
- Replacing the run-state engine. Concurrent dispatch integrates with the existing v6 run-state engine; it doesn't reinvent persistence.
- Parallelizing the validate, codex-review, or bugbot steps. Those stay sequential in this spec; future work.

## Architecture

```
Plan file (with depends_on: annotations)
         |
         v
Dependency graph extractor  --->  DAG of TaskNodes
         |
         v
Scheduler  ---  reads { concurrency.maxParallelSubagents, budgets } from guardrail.config.yaml
         |
         v
Per-task state machine: pending -> started -> completed -> merged
A downstream task is "ready to dispatch" ONLY when ALL declared and implicit deps have
reached state `merged`, not just `completed`. This guarantees the downstream worktree
branches from a feature branch that contains the dep's commits.
         |
         v
Dispatch loop:
  - dispatch up to N subagents in parallel (N <= effective concurrency, see below)
  - each subagent runs in its own worktree at .claude/worktrees/<run-ulid>/<task-id>/
  - each subagent's commits land on a per-task branch (autopilot/<run-ulid>/<task-id>),
    branched from feature/<topic-slug> at the current tip (captured as base_sha)
  - on completion: capture commit_shas via `git rev-list --reverse base_sha..task_branch`,
    emit task.completed event to run-state with exit_status
  - on failure: emit task.failed; halt the dispatch (don't start any not-yet-running tasks);
    SIGTERM in-flight subagents; do NOT auto-merge any in-flight or failed-tier tasks
         |
         v
Merge orchestrator (runs immediately after each task.completed with exit_status=success,
if and only if all of that task's deps are already in state `merged`):
  - acquire repo-level git mutex (GitOperationQueue)
  - cherry-pick base_sha..task_branch onto feature/<topic-slug> in commit order
  - on conflict: collect diagnostics first (see Conflict policy below), emit
    task.merge_conflict event with paths + index stages, THEN abort
  - on success: emit task.merged, cleanup worktree+branch
  - release mutex
         |
         v
Final state: feature branch contains all task commits in dependency order
```

**Effective concurrency** is computed at dispatch and reported in the run summary:
`effective = min(config.maxParallelSubagents, providerRateLimitConcurrency, remainingTaskCount)`.

**Invariant:** A task is only considered "merged" after its commits land on the feature
branch. Tasks in state `completed` but `not merged` remain on their task branches; their
worktrees are preserved for inspection until either merge succeeds or the user explicitly
discards them.

Three new components:

1. **`src/core/concurrent-dispatch/dep-graph.ts`** — parse plan, build DAG, detect cycles, topologically sort into tiers.
2. **`src/core/concurrent-dispatch/scheduler.ts`** — manages per-tier batch dispatch, respects `maxParallelSubagents`, accumulates per-subagent cost into the run-state budget.
3. **`src/core/concurrent-dispatch/merge-orchestrator.ts`** — cherry-picks task branches back into the feature branch in topo order; escalates conflicts.

## Annotation fallback policy

Compatibility rule for plans without `depends_on:` annotations:

- **No annotations on ANY task in the plan**: scheduler defaults to sequential execution
  (effective concurrency = 1), reproducing v7.10.0 behavior. Override with
  `concurrency.assumeIndependentWithoutDependsOn: true` to opt in to file-overlap-inferred
  parallel execution.
- **Some tasks have annotations, others don't**: scheduler uses the explicit annotations
  where present and falls back to file-overlap inference for unannotated tasks.
  Unannotated-and-non-overlapping tasks are treated as independent.
- **Cycle detection runs in all cases** before dispatch.

This ensures existing plans without annotations don't suddenly run in parallel and hit
surprising correctness issues.

## Plan format extension (depends_on)

Tasks in `docs/superpowers/plans/*.md` currently look like:

```markdown
### Task 1: Component foo

**Files:**
- Create: `src/foo.ts`
- Test: `tests/foo.test.ts`

- [ ] **Step 1: ...**
...
```

Extend the header optionally:

```markdown
### Task 3: Component baz

**Files:**
- Create: `src/baz.ts`
- Modify: `src/foo.ts:42-60`

**depends_on:** Task 1, Task 2

- [ ] **Step 1: ...**
```

Rules:
- `depends_on:` is optional. Absent = no declared deps; the scheduler will infer from file overlap.
- Refers to tasks by their `### Task N: <name>` heading; case-sensitive on the number, fuzzy on the name.
- Cycles are a hard error. Surface to user with the cycle path; do not auto-resolve.
- A task that modifies a file another task creates has an implicit dependency on the creating task; the scheduler injects it.
- File overlap (two tasks touching the same file path) without explicit `depends_on:` is a warning, not an error. The scheduler treats them as a sequential pair in topo order based on which task ID appears first in the plan. Surface this in the run report so the user can add explicit `depends_on:` if the ordering was wrong.

## Worktree lifecycle (state-based cleanup, fixed from pass 2 and 3)

For each concurrent run with run-ulid `01HZK...`:

1. **Setup (once per run, before Step 3 starts):**
   - Step 2 of the autopilot pipeline creates `feature/<topic-slug>` but does NOT check
     it out in the main worktree. Instead, it stays as a ref only, with the main worktree
     left on its prior branch (Step 2 of the autopilot SKILL.md needs updating to match).
   - Concurrent dispatch then creates the SOLE checkout of `feature/<topic-slug>` in the
     integration worktree: `git worktree add .claude/worktrees/<run-ulid>/integration/
     feature/<topic-slug>`. This is mandatory, not optional — git linked worktrees forbid
     the same branch being checked out twice, so we must own the only checkout.
   - All merge orchestrator commands run with `git -C
     .claude/worktrees/<run-ulid>/integration/`; the main repo working tree is untouched
     for the entire run.
   - Per-task worktrees are created lazily as the scheduler dispatches them, branched
     from the integration worktree's current HEAD.
2. **Per-task setup (when scheduler dispatches a task):**
   - Under git mutex: `git worktree add .claude/worktrees/<run-ulid>/<task-id> -b
     autopilot/<run-ulid>/<task-id> <current-feature-tip>` (capture base_sha)
   - Subagent runs in this worktree; commits land on `autopilot/<run-ulid>/<task-id>`
3. **Cleanup is STATE-BASED, not phase-based** (fixed from pass 1 inconsistency):
   - Only tasks that reach state `merged` have their worktrees and branches removed.
   - Tasks in ANY of these terminal-but-not-merged states keep worktree + branch:
     `failed`, `interrupted`, `completed-but-unmerged` (orphaned by upstream failure),
     `timeout`, `merge_conflict`, `ancestry_violation`.
   - Cleanup of preserved worktrees/branches is the user's call via
     `claude-autopilot runs cleanup <run-ulid>` (with explicit confirmation prompt
     listing what will be destroyed).
4. **Run-level success path:**
   - Every task is `merged` → integration worktree removed → all per-task worktrees +
     branches gone (already cleaned up state-by-state) → feature branch has all commits.

Worktree path collisions: if a prior run crashed and left `.claude/worktrees/<run-ulid>/`
behind, the scheduler refuses to start until `claude-autopilot runs gc --older-than-days
0` clears it. (Reuses existing run-state gc, now extended to check the new file lock.)

## Scheduler dispatch

Config in `guardrail.config.yaml`:

```yaml
concurrency:
  maxParallelSubagents: 3              # default 3; schema range 1..8 (above 8, API rate limits dominate)
  perSubagentTimeoutMs: 1800000        # 30 min hard timeout per subagent
  assumeIndependentWithoutDependsOn: false  # opt-in to file-overlap inference when NO task has annotations
budgets:
  perRunUSD: 10                # existing; enforced across the concurrent fleet via reservations
  perPhaseUSD: 5               # existing
  perSubagentUSD: 3            # HARD cap per task. Reject dispatch if preFlightEstimate > perSubagentUSD; SIGTERM the subagent and emit task.failed if actual_cost_usd exceeds it mid-execution. If you want a soft initial reservation instead, leave perSubagentUSD null and rely solely on perRunUSD.
```

Schema validation: `concurrency.maxParallelSubagents` is `integer`, `minimum: 1`, `maximum: 8`.
`presets/schemas/guardrail.config.schema.json` enforces this; invalid values fail at config-load.

**Budget reservation semantics (with atomic event writes):**

All event writes and budget mutations route through a single per-run **serialized event
writer** that holds an exclusive file lock on `events.ndjson` for the duration of:
`(replay current budget state) → (check remaining vs reservation) → (append event) →
(fsync) → (release lock)`. Subagents that produce telemetry mid-execution send it via
IPC to the scheduler; the scheduler is the only writer to events.ndjson.

This eliminates the race where two concurrent dispatches both pass the `budget.remaining
< preFlightEstimate` check and over-reserve.

- Before dispatch: under the writer lock, replay `budget_reserved` minus `budget_released`,
  compute `remaining = perRunUSD - reserved_total`; if `remaining < preFlightEstimate`,
  halt and emit `task.budget_halt`. Else append `task.budget_reserved` (capped by
  `perSubagentUSD`) and fsync.
- During execution: subagent telemetry streams to scheduler; scheduler accumulates per-task
  actual usage in memory; if actual exceeds reservation, scheduler appends
  `task.budget_increased_reservation` event under the writer lock (and re-checks
  `perRunUSD`; halt the run if exceeded).
- On completion: under the writer lock, append `task.budget_released` with
  `actual_cost_usd` and `delta_vs_reservation_usd`. Reservation closes.
- On failure: same — `task.budget_released` with actual cost to date.
- On resume: re-replay events.ndjson under the writer lock; reservations reconstructed
  by `reserved_total = sum(budget_reserved + budget_increased_reservation) - sum(budget_released)`.

Dispatch algorithm:

```
remaining = all tasks
while remaining:
  ready = tasks in remaining whose deps are all in state `merged` (not just completed)
  if not ready:
    if any task is in_progress or pending_merge: wait for next state transition
    else: deadlock (shouldn't happen if cycle detection ran; raise CRITICAL with full task state dump)
  for task in ready (up to effective_concurrency - in_progress_count):
    if budget.remaining_after_reservations < task.preFlightEstimate: halt, escalate
    reserve budget; emit task.budget_reserved event
    [under git-mutex] git worktree add ... + branch from current feature tip (capture base_sha)
    dispatch subagent in worktree; emit task.started event with base_sha
  await first completion or merge-eligibility transition
  on subagent exit:
    [under git-mutex] verify commits via: git rev-list --count base_sha..task_branch
    if commit_count == 0: emit task.failed with reason='no_commits'
    else: emit task.completed with exit_status, commit_shas (rev-list --reverse), actual_cost_usd
  immediately after task.completed (if all deps already merged): run merge orchestrator for this task
```

**Subagent-output detection (fixed from pass 1):**
Detect "wrote nothing" by counting commits between `base_sha` (captured at dispatch) and
the task branch tip:
```
commit_count = $(git -C <repo> rev-list --count <base_sha>..autopilot/<run-ulid>/<task-id>)
if [ "$commit_count" -eq 0 ]; then treat as failure (no_commits); fi
```

**Task branch ancestry validation (pass 2):** before accepting `task.completed`, verify:
```
git merge-base --is-ancestor <base_sha> autopilot/<run-ulid>/<task-id>
```
If this returns non-zero, the subagent rebased/reset/force-updated its branch off the
recorded base. Treat as failure (`error_type: 'ancestry_violation'`); do NOT cherry-pick.
Also reject merge commits in the task branch unless `concurrency.allowMergeCommitsInTasks:
true` (default `false`). Record the immutable `task_branch_tip_sha` in `task.completed`
and use that SHA (not the mutable branch name) for all subsequent merge/resume operations.

Separately, if the subagent left uncommitted changes in the worktree, log as a contract
violation but do not auto-commit (the subagent contract requires committing its own work).

Failure semantics:

- **Single subagent failure**: emit `task.failed`, cancel all in-flight subagents (SIGTERM
  the subagent's **entire process group** via `kill -TERM -<pgid>` to also catch child
  processes; if no response in 30s, SIGKILL the group). Do NOT dispatch any further tasks.
  Surface to user with: failed task ID, error message, list of MERGED task IDs (the only
  guaranteed-on-feature-branch set), list of COMPLETED-BUT-NOT-MERGED tasks (their commits
  exist only on task branches and worktrees are preserved), list of in-flight task IDs
  (worktrees also preserved).
- **Budget exceeded mid-run**: same as single failure but with reason "budget exceeded".
- **Subagent timeout**: SIGTERM at perSubagentTimeoutMs (process group); if no response in
  30s, SIGKILL group. Emit BOTH `task.timeout` (informational, with kill signal metadata)
  AND `task.failed` (terminal, `error_type: 'timeout'`). The dual-emission ensures resume
  classification treats it terminally.
- **No commits produced**: `git rev-list --count base..tip == 0`; emit `task.failed`,
  `error_type: 'no_commits'`.
- **Ancestry violation**: `git merge-base --is-ancestor base task_branch` returns non-zero;
  emit `task.failed`, `error_type: 'ancestry_violation'`.

## Merge orchestrator

Runs **per-task in strict plan-declaration topo order** as completions arrive. The
mergeable rule:

A task `T` is eligible to merge when ALL of these are true:
(a) `T` is in state `completed` with `exit_status=success`
(b) all of `T`'s declared and implicit dependencies are in state `merged`
(c) every task `T'` that appears BEFORE `T` in plan-declaration order AND whose deps are
    all merged is either (`merged`) or (`failed` / `interrupted` / `merge_conflict`)

Clause (c) is the strict-plan-order gate (fixed from pass 3 ambiguity). It means: if
Task 2 and Task 3 are independent siblings and Task 3 finishes first, Task 3 does NOT
merge until Task 2 reaches either `merged` (success path) or a terminal-failure state
(in which case the run is halting anyway, but Task 3 might still want to be inspected on
its preserved branch). This guarantees the final history is deterministic across re-runs
of the same plan even when subagent completion timing varies.

Implementation: the scheduler maintains a `mergeable_queue` ordered by plan-declaration
index. Each tick: examine the queue head; if the head meets (a)+(b), and clause (c)
is implicitly satisfied (everything earlier is either merged or terminal), merge it.
Otherwise wait for the next state transition.

```
acquire repo file lock (.claude/run-state/repo.lock via flock/fcntl)
acquire git-mutex (GitOperationQueue serializes ALL repo-level git ops:
                   worktree add/remove, branch create/delete, cherry-pick, abort, GC)

# Use IMMUTABLE SHAs from the task.completed event, NOT mutable branch names
base_sha = task.completed.base_sha
tip_sha  = task.completed.task_branch_tip_sha

# Re-verify ancestry under the lock (in case the branch was tampered with between
# task.completed emission and merge)
if ! git -C <integration-worktree> merge-base --is-ancestor $base_sha $tip_sha; then
  emit task.merge_aborted { reason: 'ancestry_violation_at_merge_time' }
  release locks; halt
fi

commit_list = $(git -C <integration-worktree> rev-list --reverse $base_sha..$tip_sha)
git -C <integration-worktree> cherry-pick $commit_list   # all task commits in order

if conflict:
  # CAPTURE DIAGNOSTICS BEFORE ABORT
  conflicting_paths = $(git diff --name-only --diff-filter=U)
  index_stages = $(git ls-files -u)
  porcelain = $(git status --porcelain)
  emit task.merge_conflict event with { task_id, conflicting_paths, index_stages, porcelain }
  # write conflict report to .claude/run-state/<run-ulid>/conflicts/<task-id>.md
  git cherry-pick --abort
  release git-mutex
  halt entire run; escalate with task ID, conflicting paths, link to conflict report

on success:
  emit task.merged event
  git worktree remove .claude/worktrees/<run-ulid>/<task-id> --force
  git branch -D autopilot/<run-ulid>/<task-id>
  release git-mutex
```

Cherry-pick rather than merge so that the final history is linear (matches the existing
autopilot squash-merge contract). Each task contributes its commits in order via the
recorded base_sha; this fixes the pass-1 issue where `tip~..tip` would have dropped
earlier commits in multi-commit task branches.

**Git critical-section serialization (two-layer: in-process + cross-process file lock):**

Layer 1 (in-process mutex): all repo-level git operations within a single scheduler
process acquire `GitOperationQueue`'s async mutex. Cheap, fast.

Layer 2 (cross-process file lock, AUTHORITATIVE, using flock/fcntl — NOT
check-then-create): all CLI commands that mutate repo state (`autopilot`, `run resume`,
`runs gc`, `runs cleanup`, internal merge orchestrator) call a `withRepoLock()` helper
that:

1. Opens `.claude/run-state/repo.lock` with `O_CREAT | O_RDWR`
2. Acquires an exclusive `flock(LOCK_EX | LOCK_NB)` on the file descriptor
   (or `fcntl(F_SETLK)` for portability; pick one consistently). If the lock fails
   because another process holds it, read the existing lock metadata, classify as
   live or stale (see below), and surface to user.
3. While holding the lock: truncate the file, write JSON metadata (pid, hostname,
   command, run_id, acquired_at_iso), fsync
4. Run the wrapped operation
5. Truncate the file (clears metadata), fsync, release flock via `close(fd)`

The flock guarantees atomicity: only one process can hold the exclusive lock at a time,
even across separate CLI invocations. Lock-file existence is NOT the synchronization
mechanism; the kernel-level flock is.

Stale-lock handling (only invoked when flock acquisition FAILS): read the existing
metadata. If the recorded PID is not running on the recorded hostname AND
`acquired_at_iso` is > 1 hour old, treat as stale and surface to user with: stale-lock
metadata + explicit `claude-autopilot runs cleanup --force-unlock` recovery command.
Do NOT auto-clear stale locks without user confirmation; a still-running process on
another tty could be writing.

Multi-process test: spawn two `node` processes that both call `withRepoLock()` and
verify only one enters the critical section at a time; the second blocks on flock
until the first releases.

Subagent file edits and `git commit` inside their OWN worktree do NOT acquire either lock
(they operate on private worktree refs and pack files specific to the worktree). This
avoids ref/worktree race conditions without blocking concurrent subagent work.

**Repo-state preconditions for merge orchestrator** (verified under the git mutex BEFORE
cherry-pick):
- HEAD is on `feature/<topic-slug>` (the branch from autopilot Step 2)
- `git status --porcelain` is empty (clean working tree)
- No in-progress merge/rebase/cherry-pick (`.git/MERGE_HEAD`, `.git/REBASE_HEAD`,
  `.git/CHERRY_PICK_HEAD` all absent)
- HEAD SHA matches the recorded `feature_branch_sha` from the most recent `task.merged`
  event (or, if no task has merged yet, matches the Step 2 branch-creation SHA recorded
  in the run-state run-init event)

If any precondition fails: emit `task.merge_aborted` with the violating condition; halt
the run; surface to user. Do NOT attempt to "fix" the working tree state automatically.

**Optional integration worktree** (recommended): if `concurrency.useDedicatedMergeWorktree:
true` (default `true` for safety), the merge orchestrator runs inside a dedicated
`.claude/worktrees/<run-ulid>/integration/` worktree checked out to `feature/<topic-slug>`,
so the user's main working tree state cannot interfere.

Conflict policy:

- A conflict means the dep graph missed an implicit dependency. Halt the run; do NOT
  auto-resolve, even for simple cases. The user fixes by either:
  (a) adding an explicit `depends_on:` to the plan and rerunning,
  (b) manually resolving the conflict (the worktree and task branch are preserved), or
  (c) marking the offending pair as sequential in `guardrail.config.yaml`.
- Diagnostics are persisted to `.claude/run-state/<run-ulid>/conflicts/<task-id>.md`
  BEFORE the cherry-pick abort, so users have full conflict context after the abort
  clears the working tree.

## Integration with run-state engine (v6)

Concurrent dispatch emits these new events to `events.ndjson`:

Authoritative event inventory (must match `src/core/run-state/types.ts` exactly — pass 3
flagged drift between this list and the files-to-change section):

- `task.started` `{ task_id, worktree_path, branch, base_sha, subagent_id, dispatched_at, preflight_cost_estimate_usd }`
- `task.budget_reserved` `{ task_id, reserved_usd, run_budget_remaining_after_reservation_usd }`
- `task.budget_increased_reservation` `{ task_id, prior_reserved_usd, new_reserved_usd, reason }`
- `task.budget_released` `{ task_id, actual_cost_usd, delta_vs_reservation_usd }`
- `task.completed` `{ task_id, base_sha, task_branch_tip_sha, task_branch_name, commit_shas, completed_at, actual_cost_usd, exit_status }` — `task_branch_tip_sha` is the immutable authoritative ref for all subsequent merge/resume operations; `task_branch_name` is diagnostic only
- `task.failed` `{ task_id, error_message, error_type, failed_at, actual_cost_usd }`
- `task.merged` `{ task_id, feature_branch_sha_after_merge, merged_at }`
- `task.merge_conflict` `{ task_id, conflicting_paths, index_stages, porcelain, conflict_report_path }`
- `task.merge_aborted` `{ task_id, reason, precondition_violated, occurred_at }`
- `task.timeout` `{ task_id, timeout_ms, killed_signal }` — informational; ALSO emit `task.failed` with `error_type: 'timeout'` for resume-classification terminality
- `task.budget_halt` `{ task_id, budget_remaining_usd, preflight_estimate_usd }`

Total: 11 event types.

**Resume semantics** (when `claude-autopilot run resume <ulid>` is invoked mid-Step-3):

Tasks are classified by the LAST terminal event in events.ndjson:

- `task.merged` last → DONE. Skip.
- `task.completed` (exit_status=success) last → eligible for merge. Re-run merge
  orchestrator for this task if its deps are all merged.
- `task.failed` last → terminal failure. Surface; do not retry without user intervention.
- `task.merge_conflict` last → terminal conflict. Surface conflict report path; user must
  resolve before resume can continue.
- `task.started` with NO terminal event (no completed/failed/timeout) → UNKNOWN /
  INTERRUPTED. Do NOT auto-merge any partial commits. Preserve worktree. Mark task as
  `interrupted` and require explicit user confirmation (`claude-autopilot run resume <ulid>
  --confirm-interrupted-tasks <task-id>...`) to either re-dispatch from a clean branch
  (discarding partial commits) or merge what's there (user accepts risk).
- No `task.started` event → not started; dispatch from scratch respecting deps.

This is stricter than the pass-1 draft which would auto-merge "if commits present"; that
was unsafe because commits alone don't prove the subagent finished its task contract.

## Testing strategy

Unit (fast, no git):
- Dep graph: tasks with explicit deps, tasks with file-overlap inferred deps, mixed,
  cycles, missing references, fuzzy name matching, fallback-policy decision tree
- Scheduler: deterministic merge order (plan-declaration order tie-break), batch dispatch
  with mock subagents, budget halt scenarios, timeout/cancellation, deadlock detection,
  ancestry validation rejecting tampered task branches
- Merge orchestrator: cherry-pick order with multi-commit branches (verifying ALL commits
  land, not just tip), conflict escalation with full diagnostic capture, dry-run mode,
  precondition violations (dirty tree, wrong HEAD, in-progress merge)
- Budget reservation: reserve/release/replay including resume scenarios, atomic check+reserve
  under concurrent callers, over-reservation prevention
- GitOperationQueue: in-process mutex correctness under simulated concurrent callers
- Repo file lock: PID/hostname stale detection, lock contention, force-unlock recovery

Integration (uses real git, controlled):
- 3-task plan with 1 root + 2 parallel leaves: verify both leaves run in parallel AND
  merge cleanly in plan-declaration order
- Multi-commit task branch: subagent commits 3 separate commits; verify all 3 land via
  cherry-pick base..tip in order
- Cherry-pick conflict injected: verify diagnostics persisted to
  `.claude/run-state/<ulid>/conflicts/<task-id>.md` BEFORE abort runs
- Ancestry violation: simulate subagent rebasing its branch off a different base; verify
  scheduler detects and treats as failure
- Resume scenarios:
  - `task.started` with no terminal event → classified as `interrupted`, NOT auto-merged
  - `task.completed` without `task.merged` → resume re-runs merge orchestrator
  - `task.merge_conflict` → resume surfaces conflict report, blocks until resolved
  - `task.timeout` → resume treats as terminal (dual-emission of `task.timeout` +
    `task.failed`)

Multi-process / crash scenarios (the highest-risk category per pass 2):
- Two CLI processes contend for the same repo: second process should fail with stale-lock
  error pointing at the first process's PID/host
- Process death mid-cherry-pick (with `CHERRY_PICK_HEAD` left in `.git/`): resume detects
  precondition violation and surfaces recovery instructions
- Concurrent event appends: simulated mock subagents racing to write telemetry; verify
  no NDJSON line is partially written (every line is a complete valid JSON object)
- GC refuses to run while a dispatch lock is held; force-unlock recovery works after
  manually verifying no stale process

End-to-end (slow, optional in CI):
- Full autopilot run with a small plan that has parallel tasks; verify total wall-clock
  < sequential baseline AND merged commits match expected plan ordering

## Risk classification

**High** per the autopilot risk policy (parallel git operations + cross-process locks +
auto-merge semantics + repo mutation patterns). 3 codex passes mandated. Pass 1 surfaced
4 CRITICALs, pass 2 surfaced 2 more (cross-process locking, atomic event writes), pass 3
surfaced 3 more (integration-worktree topology, immutable SHA in event schema, flock vs
check-then-create). The convergence pattern is "each pass finds real bugs in the previous
pass's design" — this is exactly what a parallel-git system looks like in design phase,
which validates the High classification.

## Out of scope (deferred)

- Parallelizing validate / codex-review / bugbot steps — separate spec, requires per-step parallelism analysis
- Distributed execution across machines — separate spec, requires distributed lock manager
- Auto-detection of deps from AST analysis — separate spec, language-specific
- Per-task model selection (e.g., Haiku for simple tasks, Opus for complex) — separate spec, requires task-complexity scoring
- Live subagent UI dashboard (showing each parallel subagent's progress in real time) — separate spec, terminal UI work

## Acceptance criteria

- [ ] A plan with 5 tasks (2 independent + 3 sequential) runs in roughly the wall-clock of the longest-chain (3 sequential), not 5x serial
- [ ] A plan with a `depends_on:` cycle fails at scheduler-init with a clear error message naming the cycle
- [ ] A failing subagent halts the run; in-flight subagents are SIGTERMed cleanly; **only** tasks in state `merged` are guaranteed present on the feature branch; completed-but-unmerged tasks have their commits on task branches and their worktrees preserved
- [ ] Budget reservations enforce `perRunUSD` correctly: a fleet of concurrent tasks whose combined reservations would exceed the cap halts before the over-reserving task dispatches
- [ ] Cherry-pick of a multi-commit task branch lands ALL commits in order (verified by counting commits before/after and inspecting log)
- [ ] Resume from a mid-flight crash: `task.merged` tasks skip; `task.completed` (no merged) tasks re-run merge; `task.started` with NO terminal event requires explicit `--confirm-interrupted-tasks` flag and does NOT auto-merge partial commits
- [ ] Cherry-pick conflict captures `git diff --name-only --diff-filter=U` and `git ls-files -u` to a persisted report BEFORE running `cherry-pick --abort`
- [ ] `concurrency.maxParallelSubagents: 1` reproduces the v7.10.0 sequential behavior exactly (backwards-compat escape hatch)
- [ ] A plan with NO `depends_on:` annotations on any task defaults to sequential execution (effective concurrency = 1); `concurrency.assumeIndependentWithoutDependsOn: true` opts in to file-overlap inference
- [ ] `concurrency.maxParallelSubagents` outside `[1, 8]` fails at config-load with a clear schema validation error
- [ ] GitOperationQueue serializes ALL repo-level git ops (worktree add/remove, branch create/delete, cherry-pick, abort) under a mutex; verified by a test that simulates concurrent callers
- [ ] Subagent-output detection uses `git rev-list --count base_sha..task_branch == 0` to detect "no commits" failures, NOT `git status --porcelain`
- [ ] Tests pass full suite (no regressions)
- [ ] `skills/autopilot/SKILL.md` Step 3 updated; `skills/writing-plans/SKILL.md` documents `depends_on:`
- [ ] CHANGELOG entry explains the new failure modes (cherry-pick conflicts halt the run; interrupted tasks require user confirmation on resume)
